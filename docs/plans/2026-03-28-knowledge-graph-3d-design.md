# 产品设计知识图谱三维主视图设计

## 背景

当前知识图谱实现正在从小规模二维拓扑视图升级到支持 `3012` 个真实实体与五类关系的图谱系统，但用户进一步明确提出知识图谱需要改为“真 3D 交互图谱”，而不是只做二维画布上的立体视觉效果。

同时，用户明确限制了首屏行为：

- 不默认展示全量 `3012` 实体的三维总图。
- 默认只展示“当前筛选结果 / 局部子图”。
- 用户通过交互逐步展开相邻节点，而不是一上来看到整张大图。

这意味着当前正在推进的“二维 dense canvas”方向不再是最终目标，知识图谱页需要切换为以 3D 场景为中心的交互模型。

## 目标

- 将知识图谱主画布升级为真 3D 场景，支持旋转、缩放和平移。
- 保留当前五类关系筛选能力：
  - 装配关系
  - 配置关系
  - 供应关系
  - 配置服务
  - 交易关系
- 首屏默认仅渲染“当前焦点节点 + 当前筛选下的局部子图”。
- 支持逐步展开邻居节点，避免一次性渲染全量 `3012` 实体。
- 保持三维布局稳定、可解释、可测试，不引入随机力导向导致的漂移。

## 非目标

- 不做全量 3012 节点同屏默认渲染。
- 不做实时力导向仿真。
- 不做沉浸式飞行、自动巡航、VR 或 WebXR。
- 不在首版实现复杂层级动画或时间轴回放。
- 不在本轮同步重构传播分析页面为 3D。

## 用户已确认的核心约束

1. 使用真 3D，而不是伪 3D。
2. 默认展示当前筛选结果下的局部子图。
3. 用户通过交互逐步展开，而不是默认展示全量总图。

## 总体方案

采用“共享图谱数据 + 局部子图派生 + 稳定三维分层布局 + 渐进展开交互”的方案。

### 核心原则

1. `WorkspaceModel` 仍然是唯一业务数据源。
2. 三维视图只是知识图谱的渲染方式变化，不单独维护一份私有图数据。
3. 3D 布局必须稳定、可解释，不依赖随机初始化。
4. 当前筛选条件决定“可见候选图”，局部子图决定“当前实际渲染图”。
5. 所有展开行为都围绕“当前焦点节点”进行。

## 图数据层次

知识图谱页中的数据需要分成三层：

### 1. 全量图谱

由 `buildKnowledgeGraphView(workspace)` 派生，包含：

- `3012` 个真实实体
- 五类关系
- 完整节点属性与边属性

### 2. 筛选后候选图

由页面筛选器生成：

- 只保留当前选中的关系类型
- 只保留这些边关联到的节点

这是“当前可被浏览的图”。

### 3. 当前局部子图

由当前焦点节点和展开状态生成：

- 默认包含当前焦点节点
- 默认包含焦点节点的一跳邻居
- 若用户展开某个节点，则将该节点的一跳邻居并入子图
- 所有节点和边都必须属于当前筛选后候选图

这是“当前实际渲染到 3D 场景里的图”。

## 焦点与展开模型

### 焦点节点

首屏焦点优先级：

1. 当前变更源组件
2. 如果该节点被筛掉，则回退到当前筛选后候选图中的第一个节点
3. 如果当前筛选后候选图为空，则无焦点

### 展开状态

页面本地维护一个“已展开节点集合”：

- 初始为空
- 双击某个节点后，将该节点加入已展开集合
- 展开节点会把其一跳邻居并入局部子图
- 点击“收起到当前焦点”时，清空已展开集合，只保留焦点的一跳邻域

### 选择与焦点的区别

- `selectedNodeId`：当前右侧详情查看的节点
- `focusNodeId`：当前局部子图围绕构建的中心节点

首版中可将两者保持一致，以降低复杂度：

- 单击节点：选中并同步切换为新焦点
- 双击节点：在选中基础上执行展开

## 三维布局设计

### 为什么不用随机力导向

随机力导向在 3D 场景里虽然视觉上炫，但会带来三类问题：

- 同一份数据每次打开布局不同，不利于演示和测试
- 节点容易遮挡、漂移，认知成本高
- 当前项目强调确定性和可解释性，与随机布局方向冲突

### 稳定 3D 分层布局

三维坐标采用稳定规则生成：

- `x / y`：沿用当前知识图谱的拓扑位置与网格排序
- `z`：表示设计域深度

建议的 `z` 分层：

- 产品节点：`z = 0`
- 产品间配置关系重点节点或次级产品层：`z = 60`
- 供应商节点：`z = 160`
- 服务关联节点可按节点域保持在供应层，但边使用独立颜色表达

这样可以让用户从任意角度看到“产品域在前、供应域在后”的三维纵深。

## 三维视觉编码

### 节点形状

- 产品节点：立方体或圆角方块
- 供应商节点：球体
- 当前焦点节点：额外发光描边或 halo
- 当前选中节点：高亮描边与悬浮标签

### 边颜色

- 装配关系：橙色
- 配置关系：紫蓝色
- 供应关系：亮蓝色
- 配置服务：绿色
- 交易关系：青色

### 标签策略

为控制信息密度：

- 默认不显示所有节点标签
- 只为以下节点显示悬浮文字：
  - 当前焦点节点
  - 当前选中节点
  - 当前悬停节点
- 展开较深时仍不显示全量标签

## 交互设计

### 3D 场景交互

- 左键拖拽：旋转场景
- 滚轮：缩放
- 右键拖拽或双指拖拽：平移
- 单击节点：选中节点并更新右侧详情
- 双击节点：展开该节点的一跳邻居

### 页面操作按钮

在右侧详情区或画布工具栏增加：

- `展开一跳`
- `收起到当前焦点`
- `重置视角`

### 筛选变化行为

当左侧关系类型筛选变化时：

- 先重建筛选后候选图
- 若当前焦点节点仍可见，则围绕该焦点重建局部子图
- 若当前焦点节点不可见，则回退到新的第一个可见节点
- 若无可见节点，则 3D 场景进入空状态

## 技术栈建议

建议引入：

- `three`
- `@react-three/fiber`
- `@react-three/drei`

原因：

- 与 React 页面状态衔接顺畅
- 便于声明式管理场景对象
- 可复用轨道控制、文字标签、辅助灯光等成熟能力

## 错误处理与空状态

### 筛选后无可见节点

- 3D 画布不渲染旧子图
- 显示空状态提示
- 保留筛选器，便于恢复

### 焦点失效

- 自动回退到新的第一个可见节点
- 如果没有可见节点，则进入无焦点状态

### WebGL 不可用

首版需保留降级策略：

- 如果 WebGL 初始化失败，回退到文本提示
- 不在本轮继续回退到二维画布
- 明确提示用户当前环境无法启用 3D 图谱

## 测试策略

### 服务层测试

需要新增：

- 局部子图派生测试
- 焦点回退测试
- 展开一跳测试
- 三维坐标映射稳定性测试

### 页面层测试

需要验证：

- 首屏只展示局部子图而不是全量图
- 单击节点会切换详情
- 双击节点会扩展局部子图
- 筛选变化后局部子图会重算
- 全部筛空后显示空状态

### 3D 组件测试边界

不测试真实 WebGL 渲染像素结果，重点测试：

- 容器是否挂载
- 交互回调是否触发
- 传入的局部子图数据是否正确
- 关键按钮与空状态是否存在

## 首版实现边界

### 首版必须实现

- 真 3D 场景
- 局部子图默认视图
- 逐步展开一跳邻居
- 五类关系筛选联动
- 选中节点详情联动
- 空状态与焦点回退

### 首版不实现

- 全量 3012 实体同屏默认渲染
- 力导向随机布局
- 传播分析页同步 3D 化
- 动态聚类折叠
- 高级动画演出

## 验证标准

- 用户进入知识图谱页后看到的是当前焦点的局部 3D 子图
- 可以旋转、缩放和平移场景
- 可以单击节点查看详情
- 可以双击节点逐步展开
- 关系筛选会真实改变当前候选图与局部子图
- 无可见节点时页面不会报错并显示空状态

## 结论

本轮最合适的 3D 方案是：

- 保留共享图谱数据与关系筛选模型
- 新增局部子图派生层
- 使用稳定分层三维坐标渲染当前局部子图
- 通过逐步展开控制复杂度，而不是默认展示全量 3012 实体总图

这样既满足“真 3D”要求，也避免把用户直接丢进不可读的超大规模三维关系云中。
