# 产品设计知识图谱扩容与关系筛选设计

## 背景

当前知识图谱页已经具备共享 `WorkspaceModel`、知识图谱派生服务和稳定拓扑布局，但图谱规模仍停留在小样本演示级别，关系类型也只有旧版的 `bom / sourcing / route` 三类边。

本轮需求要求知识图谱页具备以下能力：

- 实体总数扩展到 `3012` 个，且这些实体必须是真实图谱节点，不是只改展示数字。
- 图谱中新增并支持筛选 `5` 类关系：
  - 装配关系（产品 -> 产品）
  - 配置关系（产品 -> 产品）
  - 供应关系（产品 -> 供应商）
  - 配置服务（产品 -> 供应商）
  - 交易关系（供应商 -> 供应商）

现有“大卡片节点 + 稀疏演示数据”的实现无法承载这次规模提升，因此需要同时调整数据生成、图谱派生、筛选逻辑和画布呈现。

## 目标

- 将知识图谱扩展为 `3012` 个真实实体，并参与图谱构建、筛选和详情联动。
- 将关系类型统一升级为需求中的 `5` 类真实边。
- 在知识图谱页增加关系类型筛选区，支持按关系动态过滤图谱。
- 保持页面可读性和交互稳定性，避免 3012 个实体直接使用现有大卡片布局导致页面失控。
- 通过确定性数据生成和稳定布局保证测试可重复、演示可复现。

## 非目标

- 不做后端接口或数据库接入。
- 不做无限缩放级别的图可视化虚拟化框架替换。
- 不做知识图谱页之外的大范围业务重构。
- 不在本轮同步重做传播分析的数据模型。

## 总体方案

采用“确定性批量生成工作区数据 + 新五类关系图谱派生 + 页面级关系筛选 + 高密度拓扑画布”的方案。

### 关键原则

1. `3012` 必须是共享工作区里的真实实体总量，而不是单独写死在 UI 的展示数字。
2. 关系筛选只影响当前可见图谱，不改写底层全量图数据。
3. 图谱布局保持稳定，筛选前后节点位置尽量不跳动。
4. 画布需要针对大规模节点降密显示，不能继续沿用当前大卡片节点样式。

## 数据模型设计

### 节点规模

图谱节点总数固定为 `3012`，采用以下拆分：

- `2412` 个产品节点
- `600` 个供应商节点

### 产品节点

产品节点继续沿用现有组件语义，保留：

- `system`
- `assembly`
- `module`
- `component`

节点通过确定性规则批量生成，保证：

- 具备清晰父子结构，能形成装配关系。
- 具备跨层配置映射，能形成配置关系。
- 可与供应商形成供应关系和配置服务关系。

### 供应商节点

供应商节点继续沿用现有伙伴语义，保留：

- `supplier`
- `assembler`
- `logistics`

这样可以最大化复用现有右侧详情面板中的角色与风险展示语义。

### 关系类型映射

现有旧边类型：

- `bom`
- `sourcing`
- `route`

统一替换为以下五类：

- `assembly`
  - 产品 -> 产品
  - 表达组件、模块、装配之间的装配从属关系
- `configuration`
  - 产品 -> 产品
  - 表达产品设计中的配置依赖或选型约束
- `supply`
  - 产品 -> 供应商
  - 表达物料或组件供应关系
- `service`
  - 产品 -> 供应商
  - 表达配置、集成、调校、认证等服务关系
- `transaction`
  - 供应商 -> 供应商
  - 表达供应链伙伴之间的交易或流转关系

## 图谱派生规则

`buildKnowledgeGraphView(workspace)` 继续作为唯一图谱派生入口，但输出结构需要改造成适配五类关系的新图模型。

### 派生要求

- 所有节点都带稳定的 `id / name / kind / domain / renderLabel`。
- 所有边都带稳定的 `id / source / target / type / label / renderLabel`。
- 边方向必须与业务定义严格一致：
  - 装配、配置：产品 -> 产品
  - 供应、配置服务：产品 -> 供应商
  - 交易：供应商 -> 供应商
- 图谱派生是纯函数，同样输入始终输出同样的节点和边。

## 页面与交互设计

### 左侧概览

保留图谱概览，但统计口径调整为：

- 实体总数：固定展示全量 `3012`
- 关系总数：展示 `当前可见关系数 / 全量关系数`

同时新增关系类型筛选区，包含五个默认全选的开关：

- 装配关系
- 配置关系
- 供应关系
- 配置服务
- 交易关系

### 画布过滤规则

筛选逻辑作用于可见图谱：

- 先按选中的关系类型过滤边。
- 再保留这些边关联到的节点。
- 未被任何可见边连接的节点不进入当前画布。

这样可以保证筛选结果与画布、统计、详情完全同步。

### 空状态

如果五类关系全部取消，或筛选结果没有任何可见边：

- 画布显示空状态。
- 右侧详情显示“当前筛选下无可见节点”。
- 左侧保留筛选器和统计，方便快速恢复。

### 节点选择回退

若当前选中节点在新的筛选结果中不可见：

- 自动回退到当前可见图谱中的第一个节点。
- 若当前没有可见节点，则进入无选中状态。

## 画布呈现设计

### 为什么要改画布

当前 `TopologyCanvas` 使用大号按钮卡片渲染节点，适合几十个节点，不适合 3012 个实体。继续沿用现有样式会产生以下问题：

- 页面高度极大
- DOM 压力过高
- 标签遮挡严重
- 用户无法形成全局认知

### 新画布策略

知识图谱页改为高密度节点样式：

- 节点尺寸缩小
- 默认不为每个节点都展示完整标签
- 仅对选中节点和少量关键节点展示文本标签
- 边按五类关系使用稳定颜色区分

### 分层布局

布局仍保持稳定分层，不引入随机力导向：

- 产品节点位于左侧与中部
- 供应商节点位于右侧
- 选中源节点优先靠前显示

目标是在大规模节点条件下仍保留可解释性，并减少筛选切换时的视图跳动。

## 右侧详情设计

右侧详情仍保留当前“节点详情 + 连接关系”模式，但需适配新关系类型：

- 对产品节点显示类别、阶段、标签、参数数量。
- 对供应商节点显示位置、角色、风险、能力标签。
- 对连接关系显示：
  - 关联节点名称
  - 关系方向
  - 新五类关系标签

## 测试设计

### 服务层测试

需要新增或改写测试以验证：

- 节点总数为 `3012`
- 图谱包含全部五类关系
- 各关系方向符合定义
- 图谱派生输出稳定且可预测

### 页面测试

需要验证：

- 左侧概览显示 `3012`
- 关系类型筛选区存在五个开关
- 切换筛选后，可见关系统计同步变化
- 当前选中节点在被筛掉时能正确回退
- 画布在无筛选结果时展示空状态

## 风险与约束

### 性能风险

一次性渲染 3012 个交互节点会带来明显前端压力，因此必须在知识图谱页采用高密度渲染策略，降低单节点 DOM 成本。

### 一致性风险

如果页面层直接维护额外的假数据副本，会破坏项目现有“共享工作区模型”的设计，因此本轮必须从 `demoWorkspace` 和图谱派生服务入口统一扩容。

## 验证标准

- 知识图谱真实节点数达到 `3012`
- 五类关系均可在图谱与筛选中看到
- 关系筛选能真实改变可见图谱和统计
- 页面在 3012 个实体规模下仍可用、可选点、可查看详情
- 现有测试与新增测试通过

## 结论

本轮最合适的方案是：

- 将共享工作区中的演示数据扩展为 `3012` 个确定性真实实体
- 将旧版三类边升级为需求要求的五类关系
- 在知识图谱页加入关系类型筛选
- 将当前画布调整为适合大规模图谱的高密度呈现

这样既满足“实体真实扩容并参与展示”的要求，也保持了项目当前共享模型、稳定布局和可测试的设计方向。
